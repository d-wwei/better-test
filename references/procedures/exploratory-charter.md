# 探索性测试 Charter

触发条件：用户要求深度测试，或 strategy 发现某模块从未被深度测试

---

## 步骤

1. 定义 charter（目标 + 范围 + 时间盒 + 关注点）
2. 在时间盒内自由探索，不按脚本
3. 每个发现记录：观察 + 证据级别 + 是否需要后续
4. 时间盒结束后汇总发现，有价值的转入 test-groups 或 known-issues

## Charter 模板

```markdown
目标: 探索 [目标功能] 在 [特定条件] 下的行为
范围: [包含的模块/接口] | 不含: [排除的部分]
时间盒: [30 分钟 / 60 分钟]
关注点:
  - 边界条件: 最大值/最小值/空值/超长输入
  - 异常路径: 网络断开/超时/并发/权限不足
  - 用户易犯错: 拼写/格式/顺序错误
```

## 记录格式

```
[HH:MM] 观察: <看到了什么>
        证据: <direct / indirect>（见 investigation.md 证据分级）
        后续: 需要 → 加入 test-groups / 报 bug / 无需后续
```

## 关键原则

- 探索性测试发现新问题，自动化测试防止回归 — 互补不替代
- 时间盒是硬限制，到点就停，不管"还差一点"
- 效率数据：探索性 0.99 缺陷/小时 vs 脚本 0.24 缺陷/小时（F-Secure 研究）

方法论详解见 `methodologies/test-design.md`。
