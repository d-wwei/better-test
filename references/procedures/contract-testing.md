# 契约测试

触发条件：init Step 1 分类为 API/Web 服务且有多服务交互

---

## 什么时候用

| 场景 | 推荐 | 理由 |
|------|------|------|
| 微服务架构，多服务交互 | 强烈 | 比全链路 E2E 更稳定、更快 |
| 单体应用内部模块 | 不需要 | 单元 + 集成够了 |
| 调用第三方 API | 推荐 | 保护免受第三方变更影响 |

## 步骤

1. 识别消费者（调用方）和提供者（被调用方）
2. 消费者定义期望：调什么接口、传什么参数、期望返回什么结构
3. 将期望写成契约文件
4. 提供者验证：我确实按契约返回
5. 将契约文件纳入 test-groups.md 作为独立测试组

## 工具

- Pact（最广泛，支持多语言）
- Spring Cloud Contract（Java 生态）
- Testcontainers（可配合做集成验证）

## 关键原则

- 契约由消费者驱动（消费者定义"我需要什么"）
- 契约测试不替代集成测试 — 它验证接口结构，不验证业务逻辑
- 契约变更需双方同步

方法论详解见 `references/methodologies/design-rationale.md` 的测试设计章节。
